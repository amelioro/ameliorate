# Machine Authentication Design: Personal Access Tokens

## Purpose

This document expands Option 2 from [machine-authentication-design.md](./machine-authentication-design.md) into an implementation-focused design for app-native machine authentication via personal access tokens (PATs).

The goal is to let a machine call Ameliorate's authenticated API without browser automation or repeated cookie extraction, while still resolving the request to the same effective Ameliorate user shape that existing browser-session requests use today.

## Summary

Ameliorate should add first-party PATs that are created by a logged-in user from a new hidden settings page and then sent on API requests as `Authorization: Bearer <token>`.

The backend should validate the PAT, load the associated `User`, and build the same request context fields that current session-backed requests produce: `userAuthId`, `userEmailVerified`, and `user`.

This keeps the existing authorization model centered on `createContext` and the current tRPC middlewares, instead of introducing a second authorization model.

## Decisions For V1

These decisions are treated as resolved for the initial implementation.

- PATs authenticate API requests with `Authorization: Bearer <token>`.
- PATs are created from a browser session by a logged-in Ameliorate user on a new `/hidden-settings` page.
- PATs act with the full privileges of the associated Ameliorate user for normal API operations.
- PATs do not support scopes in v1.
- PATs do not expire by default, but the creator may optionally set an expiration date.
- PATs are stored only as secure hashes; plaintext is shown once at creation time and never stored.
- PAT management endpoints themselves require browser-session auth and must reject PAT-authenticated callers - otherwise an expiring token could be generated and used to create non-expiring tokens.
- PATs are for API authentication only. They do not replace browser login for normal page navigation.
- There is no special first-class service-user workflow in v1. If a dedicated automation identity is needed later, it can be represented as a normal Ameliorate user account that creates its own PAT through the standard session-backed UI.

## Current Architecture Constraints

The implementation needs to fit the current request-auth flow.

- [src/pages/api/trpc/[trpc].page.ts](../src/pages/api/trpc/%5Btrpc%5D.page.ts) sends all tRPC requests through `createContext`.
- [src/api/context.ts](../src/api/context.ts) currently authenticates only through `auth0.getSession(req, res)`.
- [src/api/auth.ts](../src/api/auth.ts) expects `createContext` to provide `userAuthId`, `userEmailVerified`, and optionally `user`.
- [src/api/routers/user.ts](../src/api/routers/user.ts) and other routers already rely on those context fields rather than talking to Auth0 directly.
- [src/db/schema.prisma](../src/db/schema.prisma) currently has no token table or API-key-like model.
- [src/web/common/components/UserDrawer/UserDrawer.tsx](../src/web/common/components/UserDrawer/UserDrawer.tsx) should not expose PAT management for now; the page should stay hidden behind a direct `/hidden-settings` URL.

That means the cleanest integration point is still `createContext`. PAT auth should be another way for `createContext` to resolve the same Ameliorate user shape.

## Goals

- Remove the need for manual browser-cookie extraction.
- Allow repeatable machine access without browser UI during normal operation.
- Preserve existing router authorization behavior by resolving requests to a normal Ameliorate user.
- Keep the browser-session auth path working unchanged.
- Give users a self-serve management surface for creating and revoking PATs.

## Non-Goals

- Turning Ameliorate into a full Auth0 resource server.
- Implementing OAuth delegated access, SSO reuse, or third-party app authorization.
- Adding PAT scopes or fine-grained permissions.
- Using PATs to log into browser pages.
- Adding rate limiting. This is a separate concern that can be addressed independently.
- Automating lifecycle management for dedicated service users.

Those areas may become important later, but they are not required to solve the immediate machine-authentication problem.

## Proposed Design

### Credential Format

Each PAT should be a high-entropy random bearer token generated by the backend.

- Generate 32 random bytes.
- Encode them as a URL-safe string.
- Prefix the plaintext token with an Ameliorate-specific marker such as `am_pat_` so users can recognize it easily in credential stores and logs.

One example format is:

```text
am_pat_<43-char-base64url-random-value>
```

The exact prefix string is not security-sensitive, but the token body must be cryptographically random.

Note: GitHub is a familiar example of a product that uses recognizable plaintext prefixes for token families, such as `ghp_` and `github_pat_...`.

### Storage Model

The server should never store plaintext PATs.

- On create, generate the plaintext token once.
- Compute `sha256(token)`.
- Store only the hash plus non-sensitive metadata.
- Return the plaintext token in the `create` response exactly once.

SHA-256 is sufficient here as opposed to password-hashing algorithms such as bcrypt, scrypt, or Argon2.

Those slower algorithms are designed for low-entropy human passwords that an attacker might realistically guess offline. A correctly generated PAT should already contain 256 bits of randomness, so offline guessing is not practical. In that threat model, a direct SHA-256 digest is enough, while a slow password hash would mostly add CPU cost without meaningfully improving security.

When comparing the computed hash against the stored hash during validation, consider using `crypto.timingSafeEqual` for constant-time comparison. While timing attacks against SHA-256 digests of high-entropy tokens are not a realistic threat, constant-time comparison is essentially free and aligns with defense-in-depth practice.

### Database Model

Add a new Prisma model related to `User` by `ownerUsername`.

This codebase already uses `username` as the foreign key on several user-owned tables. Prisma's default `onUpdate: Cascade` behavior for relations makes username changes workable, and using `ownerUsername` here stays consistent with the rest of the schema.

One reasonable schema shape is:

```prisma
model PersonalAccessToken {
   id               Int      @id @default(autoincrement())
   ownerUsername    String   @db.VarChar(39)
   name             String   @db.VarChar(100)
   tokenHash        String   @unique @db.VarChar(64)
   tokenPrefix      String   @db.VarChar(20)
   lastUsedAt       DateTime?
   expiresAt        DateTime?
   revokedAt        DateTime?
   createdAt        DateTime @default(now())
   updatedAt        DateTime @updatedAt

   user User @relation(fields: [ownerUsername], references: [username], onDelete: Cascade)

   @@index([ownerUsername])
  @@index([expiresAt])
  @@index([revokedAt])
  @@map("personalAccessTokens")
}
```

And extend `User` with:

```prisma
personalAccessTokens PersonalAccessToken[]
```

Field intent:

- `ownerUsername`: owning user, following the same schema pattern already used elsewhere in this repo.
- `name`: user-provided label such as `CI runner` or `local automation script`.
- `tokenHash`: SHA-256 hex digest of the full plaintext token.
- `tokenPrefix`: short non-secret identifier shown in the UI, such as `am_pat_ab12cd34`.
- `lastUsedAt`: last successful authentication time for user visibility and auditing.
- `expiresAt`: optional absolute expiry.
- `revokedAt`: soft-revocation timestamp so the token becomes unusable while retaining audit history.

## Request Authentication Flow

`createContext` should resolve auth in this order:

1. Read `Authorization` from the incoming request.
2. If it contains a Bearer token, attempt PAT authentication first.
3. If the PAT is valid, load the associated `User` and return session-equivalent context fields.
4. If no Bearer token is present, fall back to the existing `auth0.getSession(req, res)` flow.

PAT validation means all of the following are true:

- the header is in Bearer format
- `sha256(token)` matches a stored `tokenHash`
- `revokedAt` is `null`
- `expiresAt` is either `null` or in the future
- the associated `User` record still exists

### Precedence Rules

The precedence rules should be explicit.

- If a valid PAT is present, it wins over any browser session cookie.
- If a Bearer token is present but invalid, expired, revoked, or points to a missing user, do not fall back to the browser session.
- If there is no Bearer token at all, use the current session-cookie path exactly as today.

This avoids ambiguous mixed-credential behavior. It also prevents a bad PAT from silently authenticating as some other user because a session cookie happened to be attached to the same request.

### Context Shape

For valid PAT-authenticated requests, `createContext` should populate the same core fields that downstream code already expects:

- `userAuthId = user.authId`
- `userEmailVerified = true`
- `user = loaded User row`

It should also add minimal auth metadata for restricting PAT-management routes to session authentication only:

- `authSource: "session" | "pat"`

### Why `userEmailVerified` Should Be `true` For PAT Requests

PAT issuance itself already requires a logged-in Ameliorate user, and the current user-creation flow requires verified email before a user row is created.

Using `userEmailVerified = true` for PAT requests preserves existing middleware behavior without introducing a second interpretation of authentication state.

## PAT Management API

Add a dedicated tRPC router for PAT management rather than folding this into `userRouter`.

Suggested procedures:

- `personalAccessToken.list`
- `personalAccessToken.create`
- `personalAccessToken.revoke`

### Required Middleware Behavior

PAT-management endpoints should require two things:

- an authenticated, logged-in Ameliorate user
- `authSource === "session"`

That means a PAT can be used to act as a user for normal API work, but cannot be used to mint more PATs or manage existing ones.

This is an intentional security boundary. Without it, a compromised PAT could create replacement PATs and make cleanup materially harder.

### Procedure Details

#### `personalAccessToken.list`

Returns metadata only, never plaintext tokens.

Suggested response fields:

- `id`
- `name`
- `tokenPrefix`
- `createdAt`
- `lastUsedAt`
- `expiresAt`
- `revokedAt`

#### `personalAccessToken.create`

Input:

- `name`
- `expiresAt?`

Validation:

- `name` must be non-empty and reasonably bounded, for example max 100 chars
- `expiresAt`, if provided, must be in the future

Behavior:

- generate plaintext token
- hash it and store the record
- return the plaintext token once, along with the non-sensitive metadata

The plaintext value must never be retrievable again after creation.

#### `personalAccessToken.revoke`

Input:

- `id`

Behavior:

- ensure the token belongs to the current user
- set `revokedAt` if it is not already set
- return the updated metadata

Revocation should be idempotent. Re-revoking an already revoked token should not fail.

## Settings Page UX

Add a new browser-only settings page at `/hidden-settings`.

This page should be protected with the same browser-session model already used for authenticated pages such as [src/pages/notifications.page.tsx](../src/pages/notifications.page.tsx).

### Initial Page Content

For v1, the settings page only needs an API keys section.

Suggested UI elements:

- page title such as `Settings`
- section heading such as `Personal Access Tokens`
- create form with `name` and optional expiration field
- table of existing tokens with metadata and revoke action
- post-create one-time reveal panel or dialog containing the plaintext token and a copy button, with instructions to store it in a safe place and that it won't be displayed again, so you'll have to create a new token if you lose this one

The table should show enough information for a user to distinguish credentials safely:

- name
- token prefix
- created at
- last used at
- expiration
- status such as active, expired, or revoked

### Navigation

Do not add a `Settings` link to [src/web/common/components/UserDrawer/UserDrawer.tsx](../src/web/common/components/UserDrawer/UserDrawer.tsx) for now. Most users don't need to see this, so keeping it harder to find seems fine.

The existing reserved-route configuration in [src/common/reservedEndpointNames.ts](../src/common/reservedEndpointNames.ts) should reserve `hidden-settings` so the hidden page route cannot collide with a username.

## Implementation Notes

### Keep PAT Logic Out Of The Auth0 Path

PAT auth should be implemented as a separate helper used by `createContext`, not by trying to push PAT behavior into Auth0 SDK handling.

One reasonable structure is:

- new helper module for parsing and validating PAT bearer tokens
- `createContext` calls PAT helper first
- `createContext` falls back to `auth0.getSession` only when no PAT was supplied

This keeps the PAT path testable and avoids mixing app-native credentials with Auth0 SDK concerns.

### Best-Effort `lastUsedAt` Updates

`lastUsedAt` is useful, but updating it on every request would create avoidable write amplification.

The implementation should update `lastUsedAt` only when it is stale, for example when it is `null` or older than a short threshold such as 15 minutes.

That update should be best-effort rather than authentication-critical. A successful PAT auth should not fail just because the audit timestamp could not be written.

### Document Reasoning In Code

Document as many decisions + reasoning for those as possible directly next to the code that implies a decision.

Examples worth commenting locally:

- why PAT auth takes precedence over session cookies
- why an invalid PAT does not fall back to a valid session cookie
- why PAT-management routes require an interactive session via `isLoggedInViaSession`
- why `lastUsedAt` updates are throttled and best-effort

If a piece of reasoning is shared across multiple modules rather than belonging to one function, add a short summary comment block at the top of `src/common/personalAccessToken.ts`.

### Error Behavior

The PAT path should preserve current router behavior as much as possible.

- Public procedures should continue to work anonymously.
- Protected procedures should still fail through the existing middleware path when auth is absent.
- Invalid PATs should therefore result in normal `UNAUTHORIZED` behavior on protected procedures.

The main new rule is credential precedence, not a whole new error model.

### No Hard Token-Count Cap In V1

The initial implementation does not need a hard per-user PAT limit.

If token sprawl becomes a practical problem later, a limit can be added without changing the core auth design.

## Security Properties

- Database compromise does not immediately reveal usable PATs because only hashes are stored.
- Revocation is immediate for future requests because every request validates against current database state.
- Optional expiration gives users a lower-friction way to create bounded-lifetime credentials when they want them.
- PAT management requires an interactive browser session, reducing the persistence value of a stolen PAT.
- Existing authorization checks remain centralized in the current tRPC middleware chain.

## Implementation Task List

The work should be delivered in roughly this order.

1. Add the Prisma schema for PATs.
   Update [src/db/schema.prisma](../src/db/schema.prisma) with the new `PersonalAccessToken` model and `User.personalAccessTokens`, then generate and commit the migration.

2. Add shared PAT schemas and types.
   Create a new common schema module, for example `src/common/personalAccessToken.ts`, containing Zod schemas for PAT metadata, create input, and revoke input. Put any cross-cutting PAT reasoning that does not fit well next to a single implementation point in a file-level comment block here.

3. Add a PAT auth helper.
   Create a helper module under `src/api/` that parses the Authorization header, generates tokens with `node:crypto`, hashes bearer tokens, validates active PAT records, loads the associated user, and returns context-ready auth data.

4. Extend `createContext` to support PAT auth.
   Update [src/api/context.ts](../src/api/context.ts) so PAT auth is attempted before `auth0.getSession`, with explicit precedence behavior and added `authSource` metadata.

5. Add middleware for interactive-session-only operations.
   Update [src/api/auth.ts](../src/api/auth.ts) with `isLoggedInViaSession`, requiring `authSource === "session"` so PAT-management procedures cannot be called with PAT credentials.

6. Add a PAT management router.
   Create `src/api/routers/personalAccessToken.ts` with `list`, `create`, and `revoke`, then register it in [src/api/routers/\_app.ts](../src/api/routers/_app.ts).

7. Add backend tests for PAT auth and PAT CRUD behavior.
   Add tests covering valid PAT auth, invalid PAT auth, revoked PATs, expired PATs, PAT precedence over session cookies, and PAT-management routes rejecting PAT-authenticated callers. Add a dedicated test file such as `src/api/routers/personalAccessToken.test.ts` using the same `createCaller` and Prismock patterns already used in files like [src/api/routers/user.test.ts](../src/api/routers/user.test.ts). After this step, stop and wait for confirmation to proceed with front-end steps.

8. Add the settings page.
   Create `src/pages/hidden-settings.page.tsx` using the same page-auth pattern as [src/pages/notifications.page.tsx](../src/pages/notifications.page.tsx), with a section for viewing, creating, copying, and revoking PATs.

9. Keep PAT management hidden.
   Reserve `/hidden-settings` and do not add it to the user drawer yet.

10. Document API usage.
    Update [src/api/README.md](../src/api/README.md) so authenticated API examples use PATs, including a curl example with `Authorization: Bearer`.

11. Update schema documentation.
    Add the PAT table to [design-docs/database-schema.md](./database-schema.md) so the high-level schema docs stay current.

## Acceptance Criteria

- A logged-in user can create a PAT from `/hidden-settings` and see the plaintext value exactly once.
- A machine can call protected tRPC endpoints with `Authorization: Bearer <token>` and be authorized as the associated Ameliorate user.
- `isAuthenticated`, `isEmailVerified`, and `isLoggedIn` continue to work without feature-specific branching in routers.
- If both a PAT and a session cookie are present, the PAT is the credential used.
- If an invalid PAT is present, the request does not fall back to a browser session.
- Revoked PATs stop working immediately for subsequent requests.
- Expired PATs stop working immediately after expiration.
- PATs cannot be used to create, list, or revoke PATs.
- Existing browser-session behavior remains unchanged for users who do not use PATs.

## Alternatives Considered

### Manual `appSession` Cookie Reuse

This is the current approach documented in [src/api/README.md](../src/api/README.md).

It is worse than PATs because it depends on manual browser-cookie extraction, remains tied to an expiring browser-session artifact, and does not give Ameliorate a stable first-class machine-authentication contract.

### Programmatic Login To Obtain `appSession`

This avoids backend changes, but it is still fundamentally a workaround around a browser login flow.

It is worse than PATs because it stays coupled to login-page behavior, usually requires storing a real username and password, is brittle around MFA or login-flow changes, and still relies on expiring session cookies.

### Direct Auth0 Bearer-Token Support

This is the most standards-based alternative.

It is worse than PATs for the current requirement because it adds Auth0 API configuration, bearer-token validation, and claim-to-user mapping complexity that is not needed just to let trusted automation act as an existing Ameliorate user.

It may still be the right future direction if delegated third-party OAuth or cross-app SSO becomes a primary requirement.

### Auth0 Client Credentials

This identifies an application rather than a user.

It is worse than PATs because the current goal is to let a machine act as a specific Ameliorate user inside the existing authorization model. Client Credentials would still require extra mapping logic to bridge from application identity to user identity.

### Auth0 Delegated OAuth With Refresh Tokens

This is a strong standards-based design for delegated long-lived user access.

It is worse than PATs for the current requirement because it still depends on Ameliorate accepting bearer tokens directly, introduces more OAuth surface area, and solves a broader product problem than the one currently being addressed.

### Auth0 Resource Owner Password Grant

This avoids browser UI, but it requires direct handling of user credentials and still returns Auth0 tokens rather than the session shape the current API understands.

It is worse than PATs because it is both less robust and less aligned with the current architecture than simply issuing app-native credentials.

## Recommendation

Ameliorate should implement PAT-based machine authentication as the first supported long-lived machine credential.

It fits the current codebase, preserves the existing user-centered authorization model, gives users a straightforward management flow, and avoids the added platform complexity of turning the current API into a full external bearer-token resource server before that is clearly needed.

This document intentionally leaves no open design questions for the initial implementation. Later enhancements such as scopes, OAuth delegation, service-user provisioning workflows, and generalized rate limiting should be treated as separate follow-on work rather than blockers for shipping PATs.
